The seven goals

Each goal lists what is shipped on v2, what is in flight, and the residual gap to the North Star — with source citations.

1

Mimic what we practice in the real world

Partial
A single stage executed by a configurable ensemble of agents with clear responsibilities — Owner, Collaborator, Verifier — preserving the same collaboration pattern and handoff semantics across harnesses. (e.g. mob elaboration: PM = Owner, Devs/QA = Collaborators, Product Leader = Reviewer.)

The three roles map almost one-to-one onto fields that already exist on every stage:

North Star rolev2 mechanismExample
Owner (PM)lead_agent — writes the primary artifactapplication-design → architect-agent
Collaborators (Devs, QA)support_agents[] — added perspectivesfeasibility → aws-platform + compliance
Reviewer (Product Leader)reviewer — separate agent, own modeluser-stories → product-lead-agent

Shipped

  • Lead + support ensemble per stage, brought in via directive.mode. core/aidlc-common/conductor.md
  • Verifier role with two reviewer personas; the reviewer is a true separate invocation with its own context and model. core/agents/aidlc-{product-lead,architecture-reviewer}-agent.md
  • Collaboration pattern + produces/consumes/requires_stage handoff identical across Claude / Kiro / Codex; only the dispatch driver differs.

In flight

  • Reviewer wired across all harnesses (Kiro fs_write cap, trustedAgents, Codex §12a step).

Gap to North Star 2.4.0

  • Collaborators aren't independent contributors. On inline mode (every multi-agent stage shipped), a support agent is "a voice you adopt, not a subagent you dispatch" — one model speaking in multiple registers, not separate participants reasoning independently.
  • The collaboration pattern isn't a configurable knob. The shape is fixed (lead → inline support → optional reviewer loop); "pipeline / swarm / review-loop / mob" is not a per-stage selectable ensemble model.
ReadThe skeleton of the three roles is shipped and maps cleanly to the mob example — but the North Star's independent ensemble (collaborators not collapsed into the conductor's voice) and a configurable pattern are the residual.
2

Customization of Behaviour

Shipped
Encode new behaviours, policies, and constraints with minimal change (≤2 targeted changes), reusable across all harnesses without per-tool rewrites.

Shipped — goal substantially met

  • Five-layer rules stack (org → team → project → phase → stage-reserved), strict-additive, conflicts checked at admission. New constraint = 1 edit to a rule file; +1 optional sensor for a deterministic check. core/rules/ · knowledge/aidlc-shared/rules-reading.md
  • Automatic cross-harness projection: core/ → dist/{claude,kiro,kiro-ide,codex} via manifests, with rules rename and {{HARNESS_DIR}} substitution; byte-parity guarded by --check. Zero per-harness rewrites. scripts/package.ts · harness/*/manifest.ts
  • Scopes (9), sensors (4), per-agent + shared knowledge as declarative no-code surfaces. Learning loop writes a confirmed correction (and optionally scaffolds a sensor) automatically.

Gap to North Star

  • Stage-level rules (aidlc-stage-<slug>.md) — reserved layer, not yet built.
3

Adaptiveness of Workflows

Partial
Scale in (triage a SonarQube report → compact Fix→Test→PR, skipping upstream stages) and scale out (defer composing a full workflow when downstream clarity is low; decide next stage(s) at each boundary). Stage composition should not be hard-wired.

Shipped

  • Scope-driven composition with a static EXECUTE/SKIP grid; incremental scopes (bugfix, refactor, security-patch) already express a compact workflow. core/scopes/ · tools/data/scope-grid.json
  • Scale-out within Construction: units-generation emits a dependency DAG; five stages fan out for_each: unit-of-work into parallel Bolts/swarm. core/tools/aidlc-{swarm,bolt}.ts

Gap to North Star 2.3.0 — largest divergence

  • Dynamic stage composition. Composition is fixed at --init by scope; the engine does not assess accumulated context / remaining uncertainty at a boundary to decide the next stage(s). aidlc-orchestrate.ts (nextInScopeStage)
  • Scale-in from an external report. No mechanism to ingest a scan report, triage findings into auto-fixable vs human-decision, and launch a compact Fix→Test→PR run.
  • Per-unit iteration plumbing: for_each: unit-of-work emits one directive but the conductor doesn't yet iterate per unit.
4

Verifier as a true Adversary

PartialIn flight
Adversarial quality gate; verifier may use a different LLM than the producer; validates deterministically against contracts/tests/schemas/policies; self-healing loop governed by a budget (max iterations/tokens), escalating to Human-in-the-Loop with full context when exhausted.

Shipped

  • Reviewer iteration loop with budget + HITL escalation: reviewer_max_iterations (default 2); NOT-READY re-invokes the builder to the cap, then escalates to the human at the gate with unresolved findings. stage-protocol.md:885 · aidlc-stage-schema.ts:42
  • Different model than producer: reviewer personas pin modelOverride: sonnet while lead/orchestrator default to Opus (xhigh); Codex maps overrides to its slugs.
  • Deterministic sensors (advisory): required-sections, upstream-coverage, linter, type-check. Reviewer field validation.

In flight

  • Reviewer wired across harnesses.

Gap to North Star 2.2.0

  • Budget by tokens, not just iterations. Today the cap is iteration count; North Star calls for a token/iteration budget.
  • Reviewer runs domain-aware validation tools (validate-domain-model / entities / rules) against explicit contracts & schemas — designed, not built. wip/unified-feature-requirements.md (Feature 2)
  • Explicit adversarial framing — reviewer prompted to refute, not confirm.
5

Support for Cyclic, Directional Flows

Partial
Forward progression plus controlled, governed feedback loops: a verifier may send an artefact back to a producer; a downstream stage may expose gaps in an upstream design. The cycles must be directional and governed.

Shipped

  • Governed within-stage feedback loop: the reviewer↔builder iteration (Goal 4) is a bounded, directional cycle.
  • Human-initiated re-entry: Operation → next Ideation cycle via the feedback-optimization artefact and the final gate's "Start New Ideation Cycle" option.

In flight 2.0.x — loop-integrity / HITL hardening

  • Stop-hook infinite-block loop.
  • Approve has no artifact verification.
  • Stop hook incentivizes rubber-stamping.

Gap to North Star 2.5.0

  • Cross-stage backward edges. The stage graph is a strict acyclic DAG (cycles rejected at compile); a downstream stage exposing an upstream gap cannot trigger an engine-managed return to that upstream stage. requires_stage is forward-only.
6

Preserve Artefact Traceability

Partial
Artefacts flow forward and accumulate technical depth; downstream stages enrich upstream artefacts (e.g. a Domain Model later gains ER diagrams, schema, API contracts) rather than spawning disconnected new ones.

Shipped

  • Forward artefact graph via produces/consumes/requires_stage edges, with a deterministic upstream-coverage sensor checking each output references its declared upstream inputs. core/tools/aidlc-sensor-upstream-coverage.ts

Gap to North Star 2.6.0

  • True progressive enrichment — downstream stages enriching the same upstream artefact in place vs. emitting a linked-but-separate file — is partial.
  • Per-stage traceability enforcement as a deterministic JSON sensor.
  • Cross-unit discovery propagation (build-time discoveries shared across units beyond declared deps).
  • Design-vs-code artefact hygiene (keep design stages design-level; de-dupe unit-test generation).
  • ADRs as a core design artefact.
7

Organizational, Not Project-Local, Artefact Repository

In flight
A shared org knowledge layer (/org-ai-kb, /aidlc-docs) spanning multiple projects, intents, and repositories without artefacts overwriting each other; reverse-engineered knowledge versioned and discoverable across teams. Scenarios: multi-repo intent; Day-2 second intent; concurrent intents in one codebase; greenfield needing brownfield knowledge; mixed; brownfield multi-repo.

Today (v2)

  • Project-local, flat, single-intent aidlc-docs/ tree; --init --force wipes prior state. No spaces / intents / multi-repo. core/tools/aidlc-lib.ts

In flight 2.1.0 — the major effort: feature/workspace-anchor (+34 commits)

  • Spaces + intents model: aidlc/spaces/<space>/intents/<slug>-<id8>/; retire --init for auto-birth; intent/space list/switch/create verbs.
  • Multiple concurrent intents in one codebase; orchestrator offers a 2nd intent on unrelated new work.
  • Multi-repo per intent (--repos at birth; --repo threaded through worktree/swarm/bolt); per-repo code KB at aidlc/codekb/<repo>/.
  • Space-level domain knowledge, learnings→practices collapse, per-intent audit shards (no merge conflicts), crash-safe flat→namespaced migration.
  • Shared org-KB layer: knowledge and reverse-engineered artefacts live in a clonable, team-scoped org-ai-kb/<team>/ repo that compounds across intents and is shareable across teams. wip/unified-feature-requirements.md (Feature 1 — org-ai-kb)
CoverageThe workspace-anchor design (wip/unified-feature-requirements.md, Feature 1 / P0) addresses all six North Star scenarios — multi-repo, Day-2, concurrent intents, greenfield-needing-brownfield, mixed, brownfield multi-repo — and the shared org-KB layer, so Goal 7 lands wholly in 2.1.0.
