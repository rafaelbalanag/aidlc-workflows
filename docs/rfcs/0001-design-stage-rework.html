1. Summary

Rework the AI-DLC design stages so the design phase produces a coherent, ID-linked component blueprint (a stable-ID component model that downstream stages reference) instead of a scatter of disconnected markdown documents, and add an explicit inter-unit contract step so parallel teams can build against agreed boundaries.

Six stages are in scope (AD, Contracts, FD, NFR-Requirements, NFR-Design, Infra), resolving into five outcomes:

  1. Application Design → Domain Design (rename + reconceptualize)
  2. Contract Design (NEW — inception 2.8)
  3. Functional Design (rework: structured entities/rules + provider API spec)
  4. NFR Requirements + NFR Design → NFR Design (merge two stages into one)
  5. Infrastructure Design (rework: consolidate to one specification, map logical → physical)

The reworked stages originate on the feature/workflow-composition-rework branch, which uses a different runtime architecture (a flexible DAG with a workflow-composition step, different agent names, and .yaml artifact files validated by node scripts). This RFC ports the concepts and content of those stages into v2's fixed-pipeline, deterministic-engine format — it does not adopt the rework branch's runtime model.

2. Motivation

2.1 The blueprint fragments today

In v2, application-design emits five independent markdown files (components.md, component-methods.md, services.md, component-dependency.md, decisions.md). Each downstream stage (functional-design, nfr-design, infrastructure-design) reads those and emits its own fresh set of markdown files. Nothing carries a stable identity forward: a "component" named in application-design has no durable link to the entity schema in functional-design, the NFR posture in nfr-design, or the infra mapping in infrastructure-design. The result is drift — the same component is re-described, slightly differently, at every stage, and nothing mechanically ties the descriptions together.

2.2 No contract boundary for parallel work

v2 has no stage that defines the interface between units. Units are generated (units-generation), then each unit is designed and built largely in isolation, with cross-unit integration left implicit. When two units must talk, the contract is discovered late — at code-generation or integration — which is the most expensive place to discover it.

2.3 The NFR split is redundant

v2 runs nfr-requirements (capture targets) then nfr-design (design patterns) as two separate per-unit stages with two gates. In practice the requirements are already captured at a high level upstream, and the split produces two passes over the same material with a gate in the middle that rarely changes direction. One consolidated stage — measurable targets + tech-stack choices + patterns + trade-offs in a single pass — is tighter and removes a gate.

2.4 Component ≠ infrastructure

v2's application-design conflates "code you write" with "infrastructure you deploy." Databases, caches, queues, and third-party services show up as components. The rework draws a hard line: a component is a bounded piece of software with its own business logic, entities, and lifecycle; databases/caches/queues/third-party services are dependencies of components, not components. Deployment topology (monolith vs microservices) is decided at units-generation, not at design time.

3.2 The blueprint chain (Option C — stable-ID references)

The design stages share a stable-ID component model. domain-design produces the canonical components artifact once, assigning every component a stable ID (cmp-NNN). Downstream stages do not copy or re-emit components; instead each emits its own artifact whose entries reference those component IDs. The blueprint-shape sensor (§4.6) verifies that every referenced cmp-NNN resolves to a component declared in the upstream components artifact — mechanical cross-stage traceability without re-emission.

This is Option C (decided in review): it captures the load-bearing benefit of the rework branch's carry-forward model — stable IDs + mechanical "does this referenced component exist" validation — while respecting v2's single-producer artifact-vocabulary rule (a canonical name is produced by exactly one stage). We do NOT adopt the rework branch's literal file carry-forward (the single components.yaml physically copied forward and grown in place), which would require re-emitting the same canonical artifact from multiple stages and rewriting every stage body.

requirements ──> domain-design ── produces `components` (stable cmp-NNN IDs)
                      │
              units-generation (group components into units; topology decision)
                      │
              contract-design (inter-unit contracts; references cmp-NNN)  ← inception 2.8
                      │
   ┌──────────────────┴───────────── per unit ──────────────────┐
   functional-design   — entities/rules reference cmp-NNN
                      │
   nfr-design (merged) — nfr-specification references cmp-NNN
                      │
   infrastructure-design — infrastructure-specification references cmp-NNN
                      │
                code-generation

Stable-ID scheme: cmp-NNN (components, owned by domain-design), ent-NNN (entities, owned by functional-design), rule-NNN (business rules, owned by functional-design). IDs are assigned by the producing stage and referenced — never renumbered — downstream.

3.3 Artifact format decision (per this RFC)

Adopt the structured-source-of-truth idea from the rework branch, but keep v2's markdown-artifact convention so the sensor system, required-sections, and upstream-coverage checks keep working unchanged: embed the structured data as fenced ```yaml blocks inside the artifact's markdown body rather than shipping separate .yaml files.

Note this is the artifact body, not YAML frontmatter — frontmatter is reserved for stage-definition metadata (slug, produces, consumes, …); the output artifacts under aidlc-docs/ carry no frontmatter. The canonical artifact name still resolves to a single .md file on disk (per docs/reference/16-artifact-vocabulary.md); the machine-readable YAML lives as a fenced block inside it, with prose/diagram views alongside.

A new sensor validates the embedded YAML blocks (component / entity / rule shape) and the cross-stage cmp-NNN ID references — see §4.6. This gives v2 the mechanical traceability the rework branch got from physically carrying components.yaml forward, but via ID references checked by a sensor rather than file re-emission.

3.4 Per-stage detail

(1) application-design → domain-design

  • Slug change application-design → domain-design (breaking; see §5.1).
  • Concept: identify the logical building blocks (components) of the system. A component = code you write (business logic + entities + lifecycle). DBs/caches/queues/3rd-party = dependencies of components, not components. Does NOT decide deployment topology.
  • Produces: components (yaml-in-md catalogue: each component with a stable cmp-NNN ID, behaviour, dependencies, dependent-components, owned entities + attributes), domain-design-questions. Human view (mermaid + summary table) folded into the components artifact. domain-design is the sole producer of cmp-NNN IDs.
  • Consumes: requirements (required); stories, RE artifacts (optional).
  • Owner: aidlc-architect-agent. Reviewer: aidlc-architecture-reviewer-agent.
  • Phase: inception, number 2.6.

(2) contract-design (NEW)

  • Concept: define the contracts between units so teams build in parallel. A contract = the formal provider/consumer agreement (data shape, protocol, failure behaviour). Cross-unit (not per-unit).
  • Placement: inception stage 2.8, after units-generation; delivery-planning shifts to 2.9 (stays the capstone). Contracts are NOT folded into delivery-planning — different concern (contracts = inter-unit API/event/schema specs by the architect; delivery-planning = Bolt sequencing/team allocation by the delivery agent).
  • Skip rule: execution: CONDITIONAL — auto-skips when units-generation yields a single unit (no inter-unit boundary to contract). Resolved O-1.
  • Produces: contracts (one spec per inter-unit boundary — OpenAPI / AsyncAPI / shared-schema), contract-summary, contract-design-questions. Contract entries reference the cmp-NNN IDs of the components on each boundary.
  • Consumes: unit-of-work + unit-of-work-dependency (required, from units-generation); components, requirements (optional context).
  • Owner: aidlc-architect-agent. Reviewer: aidlc-architecture-reviewer-agent.

(3) functional-design (rework)

  • Concept: detail business logic per unit — entities with full attribute schemas, numbered business rules with enforcement logic, workflows, state machines. Technology-agnostic.
  • Produces: entities (structured, stable ent-NNN IDs), rules (structured, stable rule-NNN IDs), api-specification (provider-side interface for this unit), human-readable functional-spec, functional-design-questions. Entity/rule entries reference the cmp-NNN component IDs they belong to.
  • Consumes: unit-of-work + unit-of-work-story-map, components, contracts (this unit's boundaries), requirements.
  • Per-unit (for_each: unit-of-work), construction. Owner/reviewer unchanged.

(4) nfr-requirements + nfr-design → nfr-design (merge)

  • Concept: in one pass, make NFRs concrete — measurable targets + tech-stack decisions + architectural patterns + explicit trade-offs.
  • Produces: nfr-specification (one doc) + nfr-design-questions. NFR postures reference the cmp-NNN component IDs they constrain.
  • Consumes: requirements (NFR section), functional-design artifacts, components.
  • nfr-requirements is removed (its outputs are absorbed into this stage). The merged stage is self-sufficient (resolved O-5): it reads whatever NFR content requirements already carries and elicits any missing targets in its own question step. requirements-analysis is NOT changed by this RFC.
  • Per-unit, construction. Owner/reviewer unchanged.

(5) infrastructure-design (rework)

  • Concept: map logical components → physical infrastructure services; define deployment architecture. Consolidated to one specification.
  • Produces: infrastructure-specification (service mapping, compute, network, security boundaries, observability, deployment strategy) + infrastructure-design-questions. Infra mappings reference the cmp-NNN component IDs they realize.
  • Consumes: nfr-specification (required), components, contracts/functional artifacts (optional).
  • Per-unit, construction. Owner/reviewer unchanged.

4. Migration

4.1 Scope grid

nfr-requirements is removed, so its column comes out of the compiled scope grid; contract-design is added as a new inception stage and needs scope membership. Resolved membership (O-2):

  • domain-design: same scopes application-design has today — enterprise, feature, mvp, workshop.
  • contract-design: wherever units-generation runs — enterprise, feature, mvp, workshop. The single-unit auto-skip (§3.4 (2) / O-1) handles intra-scope cases where a run produces only one unit.
  • nfr-design (merged): union of today's nfr-requirements and nfr-design scope membership — enterprise, feature, mvp, infra, security-patch, workshop. (nfr-requirements carried security-patch; nfr-design did not. The merged stage keeps security-patch.)
  • functional-design: unchanged (enterprise, feature, mvp, refactor, workshop).
  • infrastructure-design: unchanged (enterprise, feature, mvp, infra, workshop).

The scope grid is regenerated from each stage's scopes: frontmatter via bun aidlc-graph.ts compile; the README/SKILL scope table is regenerated, never hand-edited.

4.2 Artifact vocabulary

Per docs/reference/16-artifact-vocabulary.md, the registry is derived from produces[]. This RFC's renames/removals are a major-version artifact-surface change:

  • Removed (from application-design + nfr-requirements): component-methods, services, component-dependency, decisions (folded into components); performance-requirements, security-requirements, scalability-requirements, reliability-requirements, tech-stack-decisions (folded into nfr-specification); business-logic-model, business-rules, domain-entities (replaced by entities/rules/functional-spec); the five infra artifacts (folded into infrastructure-specification).
  • Added: components, contracts, contract-summary, entities, rules, api-specification, functional-spec, nfr-specification, infrastructure-specification, domain-design-questions, contract-design-questions.
  • Every consuming stage's consumes[].artifact must be updated; /aidlc --doctor "Graph references" + aidlc-graph compile --check catch any miss.

4.3 Stage graph + requires_stage

  • application-design row in stage-graph.json becomes domain-design (pre-seed slug/number/name before compile).
  • contract-design is a new inception row, 2.8 (after units-generation); delivery-planning shifts to 2.9.
  • functional-design.requires_stage adds contract-design.
  • nfr-design.requires_stage drops nfr-requirements, keeps functional-design.
  • Construction renumbers: nfr-design 3.3→3.2, infrastructure-design 3.4→3.3, code-generation 3.5→3.4, build-and-test 3.6→3.5, ci-pipeline 3.7→3.6.

4.4 Tests

Structural tests pin stage counts and rosters (e.g. t01, t30 scope mapping, t32 stage-graph consistency). Removing nfr-requirements and adding contract-design keeps the total at 32 (one removed, one added) but shifts the inception/construction distribution (inception 8→9, construction 7→6). Each affected test's expected counts/membership must be updated in the same change.

4.5 Version + changelog

Renames/removals of artifact names and a stage slug are breaking — bump the minor (per the in-flight pre-1.0 policy in docs/reference/16-artifact-vocabulary.md §Stability) and add a CHANGELOG entry with the upgrade note (workflows in flight on the old stage names must restart or re-map).

4.6 New sensor — blueprint-shape (shape + ID-reference validation)

Add a sensor (blueprint-shape) under core/sensors/ that:

  1. Shape check — validates the fenced ```yaml blocks in the blueprint artifacts (components, entities, rules) have the required keys (every component carries id + behaviour + dependencies; every entity an id + attributes; every rule an id + trigger + logic).
  2. ID-reference check (Option C core) — validates cross-stage referential integrity: every cmp-NNN referenced by an entities, rules, nfr-specification, infrastructure-specification, or contracts artifact resolves to a component declared in the upstream components artifact. An orphan reference emits SENSOR_FAILED with the offending id + artifact.

It is imported via the sensors: frontmatter list on domain-design, functional-design, nfr-design, infrastructure-design, and contract-design. This is the v2-native mechanism for the carry-forward traceability (resolved O-4 + Option C), running through v2's existing TypeScript sensor + dispatch model — no parallel .yaml artifact class, no node validators.

4.7 Runner skills (hard rename)

The per-stage runner skill /aidlc-application-design is hard-renamed to /aidlc-domain-design — no deprecated alias (resolved O-3). The merged NFR stage keeps /aidlc-nfr-design; /aidlc-nfr-requirements is removed. contract-design gets a new /aidlc-contract-design runner. Runner skills are regenerated by the packager's runner-gen step from the stage graph, so this follows automatically once the stage files change — no hand-authoring.

5. Risks & Trade-offs

5.1 Slug rename is breaking

application-design → domain-design breaks any in-flight workflow whose aidlc-state.md references the old slug, plus the per-stage runner skill /aidlc-application-design. Mitigation: ship as a clean rename with a CHANGELOG upgrade note; do not alias (aliasing two slugs to one stage complicates the deterministic engine's slug lookups).

5.2 Dropping a gate (NFR merge)

Merging two gated stages into one removes a human checkpoint. For teams that relied on approving NFR targets before NFR design, the single gate now covers both. Trade-off accepted: the split rarely changed direction, and the merged stage still gates once.

5.3 Option C vs literal carry-forward

We adopt Option C (stable cmp-NNN IDs + downstream ID references validated by the blueprint-shape sensor) rather than the rework branch's literal file carry-forward (one components.yaml physically copied into each stage dir and grown in place). Option C respects v2's single-producer artifact rule (no re-emitting components from four stages), avoids rewriting every stage body, and still gives mechanical cross-stage traceability. Trade-off: the blueprint detail is spread across artifacts linked by ID rather than living in one growing file — accepted, since the linkage (not the single-file shape) is the load-bearing property.

5.4 Stage renumbering ripples into prose

Removing nfr-requirements and inserting contract-design shifts stage numbers (construction 3.2→3.6, delivery-planning 2.8→2.9). Stage numbers appear in prose — knowledge files ("Stage 2.8"), stage NOTEs (2.7/2.8 cross-refs), and the stage-protocol Bolt glossary ("one pass through stages 3.1–3.7"). All such references must be updated in the same change; a stale number is a correctness bug, not just cosmetic. (Tracked as explicit steps in the implementation plan.)

5.5 contract-design in inception

Placing contract-design as the cross-unit stage (2.8) before the per-unit construction fan-out matches the rework branch's "cross-unit boundary" intent. Risk: contracts defined before per-unit functional design may need revision once unit internals are detailed. Accepted: contracts are boundaries, not internals; the "90% right up front" framing is the explicit goal.

6. Resolved Decisions

  • O-1 (resolved: yes) — contract-design is CONDITIONAL and auto-skips for single-unit projects. See §3.4 (2).
  • O-2 (resolved) — scope membership fixed in §4.1: contract-design rides wherever units-generation runs; merged nfr-design takes the union of both NFR stages' scopes (incl. security-patch).
  • O-3 (resolved: hard rename) — /aidlc-application-design → /aidlc-domain-design, no alias; /aidlc-nfr-requirements removed. See §4.7.
  • O-4 (resolved: yes — Option C) — add the blueprint-shape sensor (§4.6) for embedded-YAML shape AND cross-stage cmp-NNN ID-reference validation. YAML embedded in the artifact body (NOT frontmatter, §3.3). Stable-ID references checked by the sensor (Option C), not literal file carry-forward (§3.2, §5.3).
  • O-5 (resolved: nfr-design self-sufficient) — requirements-analysis is left untouched. The merged nfr-design reads whatever NFR content requirements carries and elicits any missing targets in its own question step. Earlier upstream NFR capture is deferred to a separate follow-up RFC.

No open questions remain.

7. Appendix — Source mapping

Reworked stages are ported from feature/workflow-composition-rework (kiro/src/stages/<stage>/definition.md). That branch's runtime model (flexible DAG, workflow-composition, .yaml artifacts, node validators, agent names aidlc-app-architect-agent / aidlc-systems-architect-agent) is not adopted; only the stage concepts and artifact structure are ported into v2's fixed-pipeline format with v2's agent roster and reviewer mechanism, using stable-ID references (Option C) in place of the branch's literal file carry-forward.
